# 液流电池提示词设计评估报告

> **评估对象**: `hyperrag/domains/flow_battery/` 下的 4 个 prompt 模板 + `config.json` + `domain_manager.py` + `validator.py` + `operate.py` 中的 `_process_json_format_extraction` 流水线
>
> **对照基准**: `FlowBattery-Hypergraph-Construction-Proposal.md`（v0.1 征求专家意见稿）

---

## 总体评价

当前的提示词设计已经是一个**可工作的 MVP**，相比原版 Hyper-RAG 的通用 prompt 有了质的飞跃。三阶段流水线（实体 → 低阶 → 高阶）拆分得当，JSON 输出格式合理，领域上下文注入机制（`{DOMAIN_CONTEXT}`）干净。在目前的阶段，它足以跑通端到端的文档嵌入流程并产出可视化超图。

但如果对标方案文档的完整愿景，**当前实现大约完成了方案的 30-35%**。差距主要不在 prompt 文本本身，而在 prompt 背后的数据流和下游消费——好几个 prompt 里精心设计的结构化字段，在 `convert_json_entity_to_standard_format` 这一步被拍平成了字符串，使得方案中最核心的"双索引"和"结构化检索"无法实现。

下面按模块逐一评估。

---

## 1. config.json：本体定义

### 做得好的

- 7 类实体类型定义清晰，每类有 description + examples，LLM 可以理解
- CONDITION 和 METRIC 有 subtype 区分（operating/processing 和 performance/material_property），这是针对液流电池的好设计
- 4 类关系类型（COMPOSITION / OPERATION / DEGRADATION / COMPARISON）覆盖了主要场景
- `entity_fields` 定义了 value / value_min / value_max / unit，为数值型实体提供了结构
- `critical_rules` 列出了 4 条关键规则

### 与方案的差距

**差距 1：本体类型数量 —— 7 vs 8**

方案建议将 DEGRADATION 拆为两类：DEGRADATION（现象层面，"发生了什么"）和 MECHANISM（原理层面，"为什么发生"）。当前 config 将两者合并为一个 DEGRADATION，描述为"Failure phenomena AND their underlying mechanisms, combined"。

这是一个**有意的简化**，在 MVP 阶段合理。但会产生一个实际问题：下游检索时，用户问"全钒电池为什么会容量衰减"（想要机理解释）和"全钒电池有哪些失效现象"（想要现象列举）会命中同一批节点，无法区分。如果后续要做"失效归因"功能（方案中的杀手级功能之二），拆分是必要的。

**建议**：暂时保留合并方案，但在 description 中建议 LLM 在 `key_attribute` 字段标注 "phenomenon" 或 "mechanism" 作为软区分。等专家反馈后再决定是否正式拆分。

**差距 2：实体字段远不如方案丰富**

方案为每类实体定义了独立的 schema 字段（如活性物质有 formula / redox_couple / category / standard_potential_V；膜有 membrane_class / ion_exchange_type / thickness_um / IEC_mmol_g 等）。当前 config 的 `entity_fields` 是一套**通用字段**，所有类型共享同一组 name / type / subtype / value / unit / description / key_attribute。

这意味着：
- 活性物质 "VO²⁺" 的化学式、氧化还原电对、标准电位等信息只能塞进 description 或 key_attribute 的自由文本里
- 膜 "Nafion 117" 的膜类（PFSA）、离子交换类型（阳离子）、厚度（183 μm）等关键属性同样缺乏结构化字段

这直接导致了方案中"结构化检索"无法实现。你没办法查"membrane_class = PFSA 且 thickness < 200 μm 的膜"，因为这些字段根本不在数据模型里。

**建议**：这是一个**工作量较大但价值极高**的改进。可以分两步走：
1. 近期：在 entity_extraction prompt 中要求 LLM 把关键属性写入 key_attribute（已经在做了），同时在 key_attribute 的格式上做约定（如 `formula=VOSO₄; redox_couple=VO²⁺/VO₂⁺; category=redox_active`），使其半结构化
2. 中期：按方案为每类实体定义独立 schema，修改 config.json 和 prompt 模板

**差距 3：关系类型 —— 4 vs 5**

方案有 5 类超边（FORMULATION / OPERATION / DEGRADATION / MECHANISM / SELECTION），当前有 4 类（COMPOSITION / OPERATION / DEGRADATION / COMPARISON）。

映射关系是：COMPOSITION ≈ FORMULATION，COMPARISON ≈ SELECTION，区别是命名和语义范围略不同。方案的 MECHANISM 在当前版本中缺失，因为 DEGRADATION 合并了现象和机理。

这和差距 1 是同一个问题的两面。

**差距 4：缺少超边组成约束**

方案为每类超边规定了"必含顶点类型"（如 OPERATION 必含 ≥1 材料 + ≥1 条件 + ≥1 性能），当前 config 的 `hyperedge_fields` 只定义了通用字段（vertices / relation_type / description / strength），**没有约束哪些本体类型应该出现在哪类超边中**。

这意味着 LLM 可能生成一条只包含两个 CONDITION 节点的 OPERATION 超边，而 validator 不会报错。

**建议**：在 config.json 中为每类 relation_type 增加 `required_ontology_types` 和 `optional_ontology_types` 字段。validator 据此做后验校验。

## 2. entity_extraction.txt：实体抽取提示词

### 做得好的

这是整套 prompt 中**写得最好的一个**，有几个亮点：

- **Rule 1（衍生电极名独立）**：这条规则非常贴合液流电池文献实际。电极改性论文中 "GF → Bio-GF-O → Bi-Bio-GF-O" 这类命名链极常见，如果不显式规定，LLM 会把它们合并
- **Rule 2（化学身份精度）**：Bi ≠ Bi³⁺ ≠ BiCl₃ 的区分对化工领域至关重要
- **Rule 3（数值必须拆分）**：value + unit 分离是结构化检索的前提
- **Rule 4（保留原名不做规范化）**：正确地把规范化推迟到下游，避免 prompt 过于复杂
- **完整的 Example**：用铁铬电池场景给了 8 个实体的完整示例输出，覆盖了活性物质/膜/条件等多种类型

### 问题

**问题 1：缺少 confidence 字段**

方案要求每个实体输出 `confidence` 值，当前 prompt 的 JSON schema 中没有。这会影响后续的质量控制——低置信度实体应进入"待审核池"。

**问题 2：description 承载过多**

由于缺少类型特定字段（如 formula / membrane_class），很多关键信息被压进 1-2 句的 description 中。例如示例中 "PFSA cation-exchange membrane separating positive and negative compartments" 实际包含了 membrane_class（PFSA）、ion_exchange_type（cation）、和功能描述三层信息，但下游只能当字符串用。

**问题 3：key_attribute 语义不够清晰**

当前 key_attribute 的定义是 "most identifying trait"，对于 ELECTRODE 是必填的（"REQUIRED for derived ELECTRODE entities"），对其他类型是可选的。但实际上每类实体"最有辨识度的特征"差异很大：
- ACTIVE_SPECIES 的 key_attribute 应该是 redox_couple 或 category
- MEMBRANE 的应该是 membrane_class + modification
- CONDITION 和 METRIC 不需要 key_attribute（它们的数值本身就是辨识特征）

一个"万能字段"会让 LLM 输出不一致。

**建议**：将 key_attribute 拆为按类型定义的字段更好，但短期内可以在 prompt 中加一段类型特定的指导：
```
For ACTIVE_SPECIES: key_attribute should be the redox couple or category (e.g., "supporting electrolyte")
For MEMBRANE: key_attribute should be membrane_class + modification (e.g., "PFSA, commercial")
For ELECTRODE: key_attribute should be base_material + treatment chain (already required)
For CONDITION/METRIC: key_attribute is usually null
```

**问题 4：没有处理表格的策略**

液流电池文献中大量关键数据出现在表格中（如性能对比表、电解液配方表）。当前 prompt 没有专门处理表格文本的指导。PDF 解析后表格往往变成杂乱的文本，LLM 需要被引导去识别这种模式。

## 3. low_order_extraction.txt：低阶关系抽取

### 做得好的

- 4 类关系类型语义清晰，每类有液流电池领域的具体示例
- 注意到了"3+ 候选的对比应留给高阶抽取"，避免了低阶和高阶之间的语义重叠
- 示例贴切（TBCC 电极 + 热处理 + 性能）

### 问题

**问题 1：输入信息不够丰富**

当前传入低阶 prompt 的实体信息是精简版的 `entity_info`（name + type + value + unit）。但 LLM 需要看到 description 和 key_attribute 才能判断两个实体之间的关系。例如，如果只给 `{"name": "HCl", "type": "ACTIVE_SPECIES", "value": 2.5, "unit": "mol/L"}`，LLM 不知道 HCl 在这里是"支撑电解质"还是"反应物"，可能误判关系类型。

看了 `operate.py` 的代码（第 750-766 行），确实只传了 name/type/value/unit，没有 description。

**建议**：把 description 也加入 entity_info。prompt 长度会增加，但关系判断的准确性会显著提升。

**问题 2：没有引入 relation_type 对应的本体约束**

当前 prompt 没有告诉 LLM "COMPOSITION 关系通常发生在活性物质之间或电极与处理方法之间"。LLM 全靠语义理解来判断。如果加上简单的约束提示，可以减少误分类。

## 4. high_order_extraction.txt：高阶关系抽取

### 做得好的

- **Rule 5（COMPARISON 触发规则）**：这是整套 prompt 中最精巧的设计。针对液流电池文献中极常见的"系列数据对比"模式（如 BET 表面积的多电极对比），给出了明确的触发模式和完整的示例。这在通用 Hyper-RAG 中完全不存在
- `evidence_span` 字段（≤120 字的原文引用）是很好的溯源设计
- `generalization` 字段（一句话总结）对下游检索有用

### 问题

**问题 1：OPERATION 超边的指导不够**

OPERATION 是"信息密度最大、最能体现超图优势"的超边类型（方案原话），但当前 prompt 对它的描述相对笼统："under this material+condition setup, these metrics were measured"。相比 COMPARISON 有专门的 Rule 5 和详细触发模式，OPERATION 缺少类似的"识别模式"指导。

建议增加 OPERATION 的触发模式描述：
```
OPERATION trigger patterns:
  - "In this cell, [material A] was used as [electrode/membrane], [material B] as [electrolyte]..."
  - "At [current density], the cell showed [CE], [VE], [EE] of..."
  - Table rows listing multiple performance metrics for one experimental setup
```

**问题 2：cardinality 过低的风险**

当前示例中的 COMPARISON 超边有 8 个顶点（4 电极 + 4 BET 值），但没有给 OPERATION 超边的示例。实际上 OPERATION 超边的典型 cardinality 是 7-15（配方 + 膜 + 电极 + 温度 + 电流密度 + 多个性能指标），LLM 可能因为没有示例而倾向于生成低阶数的超边，从而丢失完整实验上下文。

**建议**：补充一个完整的 OPERATION 超边示例，展示一个包含 8-10 个顶点的实验上下文。

## 5. query_keywords.txt：查询关键词抽取

### 做得好的

- 三层关键词分离（theme_keywords / entity_keywords / condition_keywords）是方案中"双索引检索"的前端入口，设计得很好
- 主题词对齐到了液流电池研究方向（electrolyte design, membrane selection 等）
- 示例覆盖了三种典型查询模式

### 问题

**问题 1：condition_keywords 缺少数值解析**

当前 condition_keywords 只提取了概念（如 "temperature", "operating temperature"），但没有要求 LLM 提取数值约束。例如用户问"50 mA/cm² 下哪种膜效果最好"，condition_keywords 应该输出 `{"variable": "current_density", "value": 50, "unit": "mA/cm²"}`，而不是只输出 "current density"。

这关联到方案中的结构化检索——如果查询端不提取数值，索引端即使有结构化字段也无法匹配。

**建议**：扩展 condition_keywords 的格式：
```json
"condition_keywords": [
  {"variable": "current_density", "operator": "=", "value": 50, "unit": "mA/cm²"},
  {"variable": "temperature", "operator": ">", "value": 40, "unit": "°C"}
]
```

## 6. 代码层面的关键瓶颈

### 瓶颈 1：convert_json_entity_to_standard_format 拍平了结构化数据

这是**当前实现的最大瓶颈**。`operate.py` 第 136-169 行的 `convert_json_entity_to_standard_format` 函数把所有领域特定字段（subtype / value / unit / key_attribute）序列化为字符串列表 `additional_properties`：

```python
additional_props.append(f"subtype={entity['subtype']}")
additional_props.append(f"value={entity['value']}")
additional_props.append(f"unit={entity['unit']}")
```

这意味着下游 HypergraphDB 中，一个温度实体的数据长这样：
```
entity_name: "temperature"
entity_type: "CONDITION"
description: "Operating temperature 40°C"
additional_properties: ["subtype=operating", "value=40", "unit=°C"]
```

`value=40` 是字符串，不能做 `> 40` 的数值比较。方案中的"温度 > 40°C 且活性物质 = V(IV) 的 OPERATION 超边"这类查询**在当前数据模型下不可能实现**。

**建议**：修改标准格式，保留结构化字段为原生类型：
```python
return {
    "entity_name": entity.get("name", ""),
    "entity_type": entity.get("type", ""),
    "description": entity.get("description", ""),
    "source_id": chunk_key,
    "subtype": entity.get("subtype"),
    "value": entity.get("value"),       # 保持 number 类型
    "value_min": entity.get("value_min"),
    "value_max": entity.get("value_max"),
    "unit": entity.get("unit"),
    "key_attribute": entity.get("key_attribute"),
}
```

这需要同步修改 HypergraphDB 的 upsert 接口和 storage 层，但**没有这一步，方案中 60% 的检索功能都是空中楼阁**。

### 瓶颈 2：超边的 relation_type 没有被索引

`convert_json_relation_to_standard_format` 把 `relation_type` 保存了（第 188-189 行），但 HypergraphDB 的检索接口可能没有基于 relation_type 做过滤的能力。需要确认 storage 层是否支持。

### 瓶颈 3：没有归一化步骤

当前流水线是 `entity extraction → low-order → high-order → validate → upsert`。方案的六阶段 pipeline 中，阶段 3（归一化与锚定）在当前代码中完全缺失。这意味着 "VO²⁺" 和 "V(IV)" 和 "vanadyl" 会作为三个独立节点存在于超图中，导致检索碎片化。

Rule 4（"Preserve original names, do not canonicalize"）的意图是把归一化推迟到下游，但下游目前没有实现。

### 瓶颈 4：没有跨 chunk 聚合

方案的阶段 4（跨 chunk 聚合）在当前代码中不存在。每个 chunk 独立抽取，同一篇论文中 Methods 段的配方和 Results 段的性能被分别抽取，无法还原为一条完整的 OPERATION 超边。

## 8. 建议的下一步行动（按优先级）

**第一优先级：消除数据模型瓶颈**

修改 `convert_json_entity_to_standard_format`，保留结构化字段为原生类型。同步修改 storage 层支持结构化字段的存储和查询。这一步不做，后面所有改进都只是"表面功夫"。

**第二优先级：用真实论文做端到端测试**

在改进 prompt 之前，先用 3-5 篇真实液流电池论文（建议涵盖全钒、铁铬、有机体系各一篇）跑完整流水线，检查：
- 实体抽取的准确率和召回率
- 关系类型的分类准确率
- 超边的 cardinality 分布（是否存在退化为二元的情况）
- 数值字段的拆分成功率

这些实测数据比我在这里的理论分析更有指导价值。

**第三优先级：按实测结果迭代 prompt**

根据测试发现的问题针对性调整。可能的调整方向：
- 补充 OPERATION 超边的完整示例
- 增加表格文本的处理指导
- 在低阶 prompt 中传入完整 entity_info（含 description）
- 为 key_attribute 增加按类型的格式约束

**第四优先级：实现归一化模块**

即使是最简的版本——一个人工维护的 200 词同义词 YAML 文件 + 精确匹配——也能显著改善超图的碎片化问题。不需要一开始就做模糊匹配和 canonical ID。

---

## 小结

当前的 prompt 设计在**抽取阶段**做得相当扎实，特别是 entity_extraction 和 high_order_extraction 中的领域规则。主要的差距不在 prompt 本身，而在 prompt 产出的结构化数据在下游被丢失（standard format 拍平）以及 pipeline 中缺少归一化和跨 chunk 聚合步骤。

上一位 Claude 的方案愿景是完整的，但确实工作量很大。务实的路径是：先把数据模型的瓶颈打通（保留结构化字段），然后用真实论文做测试，再根据实测结果迭代 prompt 和 pipeline。
